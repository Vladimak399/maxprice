# Задача для Codex

Проект уже содержит базовую структуру MAX-бота для мониторинга закупочных цен.

## Цель

Проверить, доработать и довести проект до production-ready состояния для GitHub + Vercel.

Бот должен получать webhook-события от MAX, парсить сообщения операторов и отправлять короткую сводку только по:

- росту закупочной цены;
- текущей закупочной цене в базе = 0.

Снижение закупочной цены должно игнорироваться.

## Технологии

- TypeScript
- Vercel Functions
- Node.js runtime
- Vitest
- Без Python
- Без Next.js UI
- Без базы данных на первом этапе

## Что проверить

1. `npm install` проходит без ошибок.
2. `npm run typecheck` проходит без ошибок.
3. `npm test` проходит без ошибок.
4. `npm run build` проходит без ошибок.
5. `/api/max/test-parse` корректно возвращает рост закупа и скрывает снижение.
6. `/api/max/webhook` принимает mock update от MAX и формирует отчет.
7. `/api/max/register-webhook` вызывает MAX API для регистрации webhook.

## Важная логика парсера

- Идти по сообщению построчно.
- Магазин брать из самого сообщения.
- Не использовать прошлый магазин из предыдущих сообщений.
- Если магазин не найден, писать `магазин не указан`.
- Цена в скобках это цена в накладной.
- Цена после дефиса это текущая закупочная цена в 1С.
- Если новая цена больше старой, это рост.
- Если старая цена 0, это отдельная проверка.
- Если новая цена меньше или равна старой, строку игнорировать.
- Числа могут быть с запятой, точкой и пробелами: `1 137,54`.
- В тексте может быть опечатка: `отличается то текущей закупочной цены`.
- Наименование товара может содержать запятые.

## Что не делать

- Не отправлять отчеты в общий рабочий чат операторов по умолчанию.
- Не логировать токены и секреты.
- Не хранить полную переписку.
- Не подключать ИИ на первом этапе.
- Не создавать задачи. Только подсвечивать важные изменения.

## Если нужно упростить

Приоритет 1: чтобы стабильно работал `price_changes`.

Остальные режимы чатов (`returns`, `warehouse`, `repricing`, `generic`) пока могут оставаться заглушками.
